# SSE Стриминг — Исправления (2026-03-14)

## Проблемы

### 1. ❌ Step метрики показывали 0ms
- Все шаги отображались с `0ms` вместо реального времени выполнения
- Шаги не показывались в real-time, а появлялись все сразу после завершения

### 2. ❌ LLM возвращал пустой ответ после 19 минут
- Логи: `chars=0, preview=<empty>, tokens[prompt=0, completion=0, total=0]`
- Пользователь ждал 19 минут и получал пустой ответ
- Нет таймаута на LLM стриминг → запрос висел бесконечно

---

## Исправления

### ✅ 1. Исправлен порядок стриминга step событий

**Проблема**: Step события добавлялись в SSE поток **ПОСЛЕ** завершения `preparedMono`:

```kotlin
// ❌ БЫЛО (НЕПРАВИЛЬНО)
return preparedMono.flatMapMany { prepared ->
    // ...
    stepEvents  // <- К этому моменту stepSink уже закрыт!
        .concatWith(Flux.just(sourcesEvent, metadataEvent))
        .concatWith(tokenStream)
}
```

**Решение**: Используем `Flux.merge()` для **параллельного** стриминга:

```kotlin
// ✅ СТАЛО (ПРАВИЛЬНО)
val contentFlux = preparedMono.flatMapMany { prepared ->
    // ... sources, metadata, tokens
}

return Flux.merge(stepEvents, contentFlux)  // Параллельно!
```

**Результат**:
- Step события начинают стримиться **сразу** при вызове `/ask/stream`
- Пользователь видит прогресс в real-time (STARTED → IN_PROGRESS → COMPLETED)
- Метрики `durationMs` корректно отображаются

---

### ✅ 2. Добавлены таймауты для LLM и подготовки контекста

**Проблема**: Нет таймаута → LLM может висеть бесконечно

**Решение**:

1. **Таймаут на подготовку контекста** (150 секунд):
   ```kotlin
   val preparedMono = Mono.fromCallable {
       ragService.prepareContextWithProgress(...)
   }
       .timeout(Duration.ofSeconds(150))
   ```

2. **Таймаут на LLM стриминг** (180 секунд):
   ```kotlin
   val tokenStream = promptSpec
       .stream()
       .content()
       .timeout(Duration.ofSeconds(180))
       .onErrorResume { e ->
           Flux.just("⚠️ Ошибка генерации ответа: ${e.message}")
       }
   ```

**Результат**:
- Если Ollama зависает → пользователь видит ошибку через 3 минуты вместо 19 минут
- Если LLM не может сгенерировать ответ → fallback сообщение вместо пустого ответа

---

### ✅ 3. Улучшена обработка ошибок

**Добавлено**:
- Логирование неудачных `tryEmitNext()` вызовов
- `.cache()` на `preparedMono` для переиспользования результата
- `onErrorResume` для LLM стриминга с понятным сообщением об ошибке
- Проверка `emitResult.isFailure` при отправке step событий

## Дополнительные улучшения (опционально)

### Если Ollama часто зависает

1. **Уменьшите контекст** (см. `RAG_OPTIMIZATION.md`):
   ```yaml
   docgen.rag.max-context-chars: 8000  # вместо 12000
   ```

2. **Используйте более легкую модель**:
   ```yaml
   spring.ai.ollama.chat.options.model: qwen2.5:7b  # вместо 14b
   ```

3. **Проверьте ресурсы Mac mini**:
   ```bash
   # Проверка памяти модели
   ollama show qwen2.5-coder:14b

   # Если не хватает VRAM → используйте квантизацию
   ollama pull qwen2.5-coder:14b-q4_K_M  # 4-bit квантизация
   ```
